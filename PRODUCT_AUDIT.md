# LifeBookAI — Продуктовая ревизия

> Этот документ — НЕ список багов и не security-аудит. Это попытка ответить на один вопрос:
> **«Почему то, что выходит из бота, не похоже на автобиографию, и что нужно изменить, чтобы это исправить?»**
>
> Минорные технические замечания (дубликат импорта, webhook-секрет, статичные роуты PDF) лежат отдельно. Здесь — только то, из-за чего пользователь уйдёт через 4 недели.

---

## 1. Что это за продукт

**LifeBookAI** — Telegram-бот, который раз в неделю принимает от пользователя текст или голосовое («что было на этой неделе») и превращает каждый вход в одну страницу личной автобиографической книги.

### Целевой пользовательский цикл

```
/start
  → онбординг (цель писательства, голос книги, частота напоминаний, день/время)
  → интейк-интервью (7 вопросов про жизнь ДО этого года)
  → AI пишет ПРОЛОГ книги (5 страниц)
  → еженедельно: пользователь присылает момент → 0–2 уточняющих вопроса от AI → AI пишет одну страницу → карточка-PNG в чат
  → каждые 5 недель: пересборка «биографического брифинга» (lifeContext)
  → milestone'ы 1/3/10/25/52: AI генерит обложку
  → milestone 3+: AI предлагает название книги
  → в декабре (или по запросу Pro): сборка PDF-книги
```

### Платная модель

- **Free:** 4 еженедельных записи (≈ месяц).
- **Pro (290 ⭐ / месяц или 2900 ⭐ / год):** безлимит, AI-обложка, AI-название, PDF-книга в декабре, «приоритет генерации».

### Ключевые сущности данных

| Модель | Роль |
|---|---|
| `User` | юзер + writingStyle + lifeContext (200-словный prose-брифинг) + reminders |
| `Entry` | сырой вход (text или voice transcript) |
| `Page` | литературная страница, 1:1 с Entry, kind ∈ {WEEKLY, PROLOGUE} |
| `ClarificationQuestion` | до 2 уточнений между вводом и финальной генерацией |
| `Memory` | извлечённые факты {PERSON, PLACE, THEME, GOAL, FEAR, …}, category ∈ {INTAKE, EXTRACTED} |
| `Book` | контейнер; обложка, AI-title, PDF |
| `Chapter` | задумывалось как синтезированная глава из нескольких Page; **по факту почти не используется** |

## 2. Архитектура написания книги (как это работает СЕЙЧАС)

Это критическая часть. Поняв её, становится очевидно, **почему результат не складывается в автобиографию**.

### 2.1 Что AI получает на вход для генерации одной страницы

Файл: [packages/ai/src/generateEntry.ts](packages/ai/src/generateEntry.ts), [packages/ai/src/prompts.ts:36-106](packages/ai/src/prompts.ts#L36-L106)

При генерации Page #N модель видит:

| Поле | Что в нём |
|---|---|
| `rawEntryOrTranscript` | Текущий ввод пользователя (полностью) |
| `lifeContext` | **200-словный prose-резюме**, пересобирается раз в 5 недель |
| `recentEntries` | 4 последние страницы — **только title + tags + daysAgo + опц. quote** |
| `memories` | До 12 фактов вида `{type, title, content}` |
| `entryNumber` | Номер записи |
| `language` | Язык |

### 2.2 Откуда берётся `lifeContext`

Файлы: [apps/bot/src/services/lifeContextService.ts](apps/bot/src/services/lifeContextService.ts), [packages/ai/src/summarizeLifeContext.ts](packages/ai/src/summarizeLifeContext.ts)

Раз в 5 недель (см. [weeklyEntry.ts:362-368](apps/bot/src/conversations/weeklyEntry.ts#L362-L368)) вызывается `refreshLifeContext`, который кормит модель:

- intake-memories (одно-двух-предложные факты от 7 вопросов онбординга),
- до 30 extracted-memories (одно-двух-предложные факты),
- 12 последних страниц — **только title + tags + mood**.

И просит сжать в **прозу ≤ 200 слов**.

### 2.3 Откуда берётся `Memory`

- **INTAKE memory:** один LLM-вызов на каждый из 6 интейк-ответов (Q7 «голос» уходит в `User.writingStyle`). Извлекает ≤ 6 структурных фактов с `confidence: 0.95`.
- **EXTRACTED memory:** ≤ 4 фактов внутри JSON-выхода `generateEntry`, эмитятся самой моделью при генерации страницы.

### 2.4 Откуда берётся пролог

[apps/bot/src/conversations/onboarding.ts:181-300](apps/bot/src/conversations/onboarding.ts#L181-L300) + [packages/ai/src/generatePrologue.ts](packages/ai/src/generatePrologue.ts)

- 5 страниц по жёстким брифам: ORIGIN → INFLUENCE → TURNING_POINT → COMPANIONS → THRESHOLD.
- Каждая следующая страница видит **полные тела** предыдущих страниц пролога.
- На выходе — один Page kind=PROLOGUE на каждую из 5 страниц.

### 2.5 Что AI **НЕ получает** при генерации еженедельной страницы

**Этот список — суть проблемы.** Когда модель пишет страницу #20, она НЕ видит:

- 🚫 Текст ни одной из 19 предыдущих написанных страниц.
- 🚫 Текст ни одной из 5 страниц пролога.
- 🚫 Сырой ввод пользователя за прошлые недели.
- 🚫 Историю диалога. Уточняющие ответы за прошлые недели.
- 🚫 Структуру времени — где сейчас в году находимся, какой это месяц книги.
- 🚫 Никаких «арок» — какие персонажи появлялись, как они эволюционировали, какие темы возвращаются.

Что модель **знает** при написании страницы #20:
- 200-словное стародавнее описание пользователя «общими словами»
- 4 заголовка предыдущих страниц + теги + 1 цитата
- ≤ 12 строчек «тип–имя–фраза» из памяти

**Это всё.** На основании этих вводных модель должна написать что-то, что выглядит как продолжение длинной автобиографии.

### 2.6 Карточка-PNG (то, что пользователь реально видит)

Файлы: [packages/renderer/src/renderEntryCard.ts](packages/renderer/src/renderEntryCard.ts), [packages/renderer/src/text.ts](packages/renderer/src/text.ts)

- SVG → PNG через `@resvg/resvg-js`.
- Размер: 1080×1920 (Stories-формат).
- Body: `font-size="22"`, `BODY_LINE_HEIGHT = 36`, `BODY_MAX_LINES = 30`, `BODY_WRAP_CHARS = 56`.
- Заголовок: 54-64pt.
- Цитата: 24pt italic.
- Шрифт: `Georgia, 'Times New Roman', serif`.
- При превышении 30 строк текст **обрезается с «…»** ([text.ts:40-42](packages/renderer/src/text.ts#L40-L42)).
- Бот шлёт ТОЛЬКО фотографию + короткую подпись (`Запись N из 52`). **Тело страницы не отправляется текстом** ([weeklyEntry.ts:127-130](apps/bot/src/conversations/weeklyEntry.ts#L127-L130)).

## 3. КОРНЕВЫЕ продуктовые проблемы

Не «было бы неплохо», а **«пользователь сейчас не получает того, за что платит».**

---

### 🔴 Проблема №1. Получается 52 разрозненных вижнетки, а не автобиография

**Это самая большая проблема. Всё остальное вторично.**

Системный промпт [prompts.ts:9-15](packages/ai/src/prompts.ts#L9-L15) утверждает:

> «You are NOT a fresh listener every week. The book is one ongoing biography, not 52 disconnected vignettes. You remember the user's background — where they're from, who matters to them, the year's recurring threads — because that's in your briefing. When the new entry resonates with something from their background or earlier pages, quietly weave a specific echo: name the same person, the same place, the same recurring fear or hope.»

**Это обещание модели, которое архитектура не даёт выполнить.** Чтобы «quietly weave a specific echo», нужен доступ к ПРОЗЕ предыдущих страниц. У модели его нет — только заголовки и теги.

#### Доказательства из кода

| Что должно работать | Реально работает |
|---|---|
| «Помнит, что в записи #4 пользователь рассказал о ссоре с мамой» | Видит только: `"4d ago — \"Тихая среда\" (мама, кухня)"` |
| «Знает, что в феврале пользователь начал бегать» | Знает: `["бег", "утро"]` где-то в `tags`, но не сцену |
| «Подхватывает интонацию ранее найденной строки» | Не видит ни одной строки прошлого |
| «Развивает арку отношений со старшим братом» | Не отслеживает арки никаких персонажей |
| «Чувствует, что неделя резонирует с турнирной точкой пролога» | Не видит ни одного предложения из пролога |

#### Почему так получилось

Архитектура построена вокруг идеи: «факты в Memory + 200-словный брифинг = достаточно для континуальности». Это неправильное допущение. **Автобиографическая проза — это не граф фактов, это голос, образы и сцены.** Голос невозможно восстановить из тегов.

200-словный `lifeContext`, который и так уже сжат ИЗ заголовков (без прозы), теряет ровно ту фактуру, которая делает мемуар мемуаром.

#### Что пользователь видит на практике

К записи #15 пользователь начнёт замечать:
- Каждая запись «начинается заново» — как будто бот не помнит ничего конкретного.
- Имя жены упоминается то полным именем, то «один близкий человек», то иначе.
- Нет ощущения «нарастающей истории», только «цепочка дневниковых заметок».
- Если в записи #5 был дождь и боль, а в #20 снова дождь — никакой переклички.

К записи #30 пользователь подумает: **«это просто красиво оформленный дневник, а не книга».**
И вот здесь он отписывается.

#### Что нужно изменить (по сложности возрастания)

**Уровень 1 — Sliding window прозы (1-2 дня работы):**
Добавить в `generateEntry` поле `recentBodies` — полные тела 2 последних страниц + 5 страниц пролога. Это сразу даёт модели голосовую и фактологическую опору.
Бюджет токенов: 2 страницы × ~800 токенов + 5 страниц × ~600 токенов ≈ 4500 токенов. У `gpt-4.1` это окей.

**Уровень 2 — Семантический ретривер (3-4 дня):**
Хранить эмбеддинги body каждой Page (`pgvector` или внешний — `text-embedding-3-small` дёшев). При генерации страницы #N: семантическим поиском по входу пользователя достать 3 самых релевантных предыдущих страницы и подать их телами. Это даёт «specific echo» — модель действительно может ответить «я уже писал об этом 17 недель назад».

**Уровень 3 — Story-state, отдельно от Memory (1 неделя):**
Ввести `NarrativeThread` модель: одна запись = одна «сюжетная нить». Поля: `userId`, `title` («отношения с отцом», «возвращение к рисованию»), `state` (running 200-300 word summary that EVOLVES). После каждой страницы фоновый job обновляет нити: «что новое мы узнали об этой нити?». При генерации следующей страницы — feed только активные нити. Это даёт ощущение арки.

**Уровень 4 — Two-pass generation (1-2 дня):**
Перед генерацией прозы — отдельный «planning» вызов: модель определяет «эта запись новая нить, продолжение, или резонанс?». Этот план идёт во второй вызов вместе с прозой. Двухэтапка надёжнее одного «сделай всё».

**Уровень 5 — Главы, не только страницы (2-3 недели):**
Каждые 4-5 страниц фоновый процесс пишет «Bridge / Chapter Intro»: 150 слов, которые связывают эти страницы темой. Эти бриджи становятся настоящими `Chapter`-записями (модель в схеме УЖЕ есть, но не используется!). На web-превью и в PDF читается как `Chapter 1 → 5 страниц → Chapter 2 → 5 страниц...`. Тогда это книга.

> **Минимум для исправления:** Уровни 1+2. Это превратит «дневник» в нечто, что субъективно ощущается как продолжающаяся история.

### 🔴 Проблема №2. Карточка-PNG плохо читается и обрезает текст

Это **второй главный отток** — не из-за смысла, а из-за UX доставки.

#### Что измеряется

[renderEntryCard.ts:62-65](packages/renderer/src/renderEntryCard.ts#L62-L65):
```
BODY_LINE_HEIGHT = 36
BODY_MAX_LINES = 30
BODY_WRAP_CHARS = 56
```

Максимум: 30 строк × 56 символов = **1680 символов на карточке**.

Сравним с тем, что модели разрешено сгенерить: системный промпт говорит [prompts.ts:73](packages/ai/src/prompts.ts#L73) «220–360 words for body». 360 русских слов — это **2000-2200 символов**.

**Любая запись на верхней границе разрешённой длины обрезается.** Полная сцена пользователя теряется в `…`.

#### Размер шрифта в Telegram

Карточка 1080×1920 PNG, в Telegram-чате на смартфоне отображается как ~340-450 px шириной (зависит от устройства). При body=22pt на 1080-wide canvas, в превью это:

```
22 × (340/1080) ≈ 6.9 пикселей
```

**6-7px эффективный размер.** Times-style серифы при таком размере становятся «месивом» — особенно для кириллицы. Чтобы прочитать, пользователь обязан тапнуть для full-screen.

#### Дополнительные проблемы рендера

1. **`wrapText` не делает переносов слов** ([text.ts:14-45](packages/renderer/src/text.ts#L14-L45)). При 56 символах строки и среднем русском слове 6-7 символов получаются строки-сироты. Нет hyphenation, нет justification.
2. **`wrapText` лишён логики абзацев** — все `\n\n` заменяются `\s+` ([renderEntryCard.ts:69](packages/renderer/src/renderEntryCard.ts#L69)). Сцена «спрессовывается» в монолит. Авторская ритмика, на которой настаивает системный промпт («Vary sentence rhythm. Short sentence, then a longer one, then a quiet beat»), визуально пропадает.
3. **Цитата позиционируется математически** через `Math.max(bodyEndY + 80, CARD_H - 240)` ([renderEntryCard.ts:86](packages/renderer/src/renderEntryCard.ts#L86)). Если тело короткое — большая пустая дыра между телом и цитатой. Если длинное — цитата налезает на тело.
4. **Bold-заголовок 54-64pt** при body 22pt создаёт перевёрнутую иерархию: заголовок огромный, текст сцены — мелкий. Книга так не выглядит.
5. **Тело и в чат-сообщение НЕ дублируется** ([weeklyEntry.ts:127-130](apps/bot/src/conversations/weeklyEntry.ts#L127-L130)). Единственный путь прочитать страницу — открыть фото full-screen или зайти в web-превью.

#### Что нужно изменить

Проблема архитектурная: **PNG-карточка не должна быть основным каналом доставки прозы.**

**Правильное разделение ролей:**

| Артефакт | Роль | Длина |
|---|---|---|
| **Карточка-PNG** | Объект для шеринга / обложка эмоции / архив | заголовок + цитата + 3-5 строк тизера сцены |
| **Текст в чате** | Основное чтение | full body, разбит на 2-3 сообщения, разделители абзацев сохранены |
| **Web-превью** | Полный «лист книги» | без ограничений длины, с хорошей типографикой (LivingBook.tsx уже неплох) |

Конкретно:
1. Сделать карточку плакатной — крупный заголовок + цитата + 3-4 строки начала сцены + «Прочитать целиком →» в подписи. Body=44-52pt, заголовок=72pt.
2. Полное тело — отдельным `ctx.reply(body)` с моноширинной структурой абзацев. Telegram нормально рендерит длинный текст.
3. Сохранить кнопку «Открыть мою книгу» — для full-screen чтения (web).
4. Если хочется один артефакт — сделать многостраничную PNG-карточку (3:4 первая, 3:4 вторая «продолжение») и слать обе.

### 🔴 Проблема №3. Интейк (онбординг) собирает слишком мало материала для пролога

[apps/bot/src/services/intakeService.ts:15-51](apps/bot/src/services/intakeService.ts#L15-L51) — 7 вопросов:

```
1. Откуда ты родом?
2. Кто на тебя сильнее всего повлиял?
3. Какой важный поворот случился ДО этого года?
4. Кто рядом с тобой сейчас? Назови 3-5 ключевых людей.
5. Что ты сейчас проживаешь?
6. Чего ты больше всего боишься потерять, о чём мечтаешь?
7. Каким голосом писать твою книгу?
```

Из этих 7 ответов (Q7 — это просто стиль, до ≤ 200 символов) AI должен:
- извлечь все INTAKE memories (≤ 36 фактов теоретически, на практике 8-15),
- написать ПРОЛОГ КНИГИ (5 страниц, 200-280 слов каждая = 1000-1400 слов),
- задать `lifeContext`,
- задать «foundation» для всех следующих 52 страниц.

#### Почему этого мало

Реальный мемуар-райтер ([Jane Friedman, NCW, MasterClass](https://janefriedman.com/2-methods-for-structuring-your-memoir/)) перед началом проводит:
- timeline mapping (10-15 ключевых событий с привязкой ко времени),
- character map (5-10 человек с описанием по полстраницы каждый),
- thematic identification (что СВЯЗЫВАЕТ эти события).

LifeBookAI собирает ~5 минут болтовни голосом и пытается из этого вытянуть 5-страничный пролог. Системный промпт пролога [generatePrologue.ts:65-67](packages/ai/src/generatePrologue.ts#L65-L67) предупреждает себя:

> «If the material is sparse, the page is honest and short; do not pad with invented detail.»

И именно так и происходит — пролог часто выходит «честно бедным», ровно потому что материала не хватает.

#### Что нужно изменить

**Вариант A — оставить онбординг коротким, но сделать «накопительный».**
- Сейчас 7 вопросов одной сессией.
- Лучше: 3-4 базовых на онбординге → пролог пишется не сразу, а через 2 недели, после ещё 5 «фоновых» вопросов, разбросанных между weekly-flow.
- Пользователь не выгорает, материала больше, пролог получается через 2 недели «как награда».

**Вариант B — углубить интейк-вопросы и собирать сцены, а не факты.**
- Q1: «Расскажи ОДНУ конкретную сцену из детства, которая до сих пор приходит в голову» (а не «откуда ты родом» — это уже факт).
- Q2: «Опиши момент, когда X на тебя повлиял — что они сказали, в какой комнате, какой был час дня».
- Сцены гораздо более кормят прозу, чем факты.

**Вариант C — дать пользователю просмотреть/исправить пролог.**
- Сейчас пролог печатается, и пользователь либо принимает, либо переделать нечего.
- После генерации — кнопки «Эта страница ОК / Перепиши / Я добавлю детали».
- Если «добавлю детали» — пользователь дописывает в чат, новая правка идёт в ту же страницу.

**Вариант D — позволить голос-only интейк (длинное голосовое 5-10 мин)** вместо 7 коротких ответов. Старый способ интервью. Транскрибировать → на вход интейк-extraction. Многим людям проще «рассказать про детство» одним потоком.

### 🔴 Проблема №4. Голос книги (`writingStyle`) не калибруется и не контролируется

Q7 онбординга («каким голосом писать?») сохраняет одну строку до 200 символов в `User.writingStyle`. Эта строка передаётся в промпт и пролога, и будущих страниц.

**Проблемы:**

1. **Без эталонов модель интерпретирует строку каждый раз заново.** На странице #5 «спокойным, ироничным» может быть похоже на Бродского, на странице #15 — на стендап-коментарий. Согласованности нет.

2. **Нет проверки тонального дрейфа.** Если случайно одна страница вышла слишком драматично, следующая страница будет генерироваться без знания об этой ошибке.

3. **Pro-фичи «more dramatic / less dramatic / more literary»** не закрепляют выбор. Нажал «менее драматично» → страница перегенерилась → следующая страница опять может быть драматичной, потому что `writingStyle` не обновился.

4. **Не объясняется пользователю, что значит каждый стиль.** «Calm, ironic, direct» — это абстракция. Лучше: предложить 4-5 готовых profiles с примером 2-3 предложений каждого.

#### Что нужно изменить

- При выборе голоса показывать тот же абзац (например, «утро, кухня, окно») в 4 разных стилях. Пользователь выбирает по примеру, а не по абстрактному слову.
- Хранить выбранный пример и явно подавать его как **few-shot пример** в каждый промпт генерации страницы. Модель тогда видит «вот РЕАЛЬНЫЙ образец того, как должен звучать этот пользователь».
- После каждых 5 страниц — фоновый «style auditor» (LLM-вызов): сравнивает свежие страницы с эталоном, флажит дрейф, возвращает «recalibration note» в `lifeContext`.

### 🟠 Проблема №5. `Memory` собирается слабо и используется ещё хуже

#### Сбор

В `EntryOutput` модель эмитит до 4 memories со схемой `{type, title: 2-8 слов, content: 1-2 предложения}` ([prompts.ts:91-94](packages/ai/src/prompts.ts#L91-L94)).

Но:
- **Текстура теряется.** Если в сырой записи 3 абзаца про маму с конкретными словами «она достала из шкафа альбом 1995 года», в memory попадёт `{type: PERSON, title: "мама", content: "тёплая, иногда обижается"}`. Альбом 1995-го забыт навсегда.
- **Нет дедупликации.** Каждая запись, упоминающая маму, может создать новую memory с `title="мама"`. К записи #20 будет 8-10 memories про маму — все короткие, все избыточные.
- **Нет эволюции.** Реальные отношения с человеком меняются. Memory `{title: "мама", content: "тёплая"}` не учитывает что 5 недель назад они поссорились.

#### Использование

`pickMemoriesForEntry` ([memoryService.ts:83-128](apps/bot/src/services/memoryService.ts#L83-L128)) — **bag-of-words keyword overlap**, фильтр на длину >3 символов. Это:
- Не находит синонимов (если в записи «бабуля», memory `{title: "бабушка"}` не зацепится).
- Не находит семантические связи (запись про одиночество не вытащит memory про потерю друга).
- Не учитывает контекст диалога (если предыдущая страница уже использовала эту memory, она опять будет в топе).

#### Что нужно изменить

- **Дедупликация по entity**: при эмиссии memory с типом=PERSON и title, нормализованным похожим на существующее → обновлять `content` через LLM-merge, а не создавать дубль.
- **Эмбеддинговый retrieval**: `text-embedding-3-small`, pgvector. Не пытаться писать «литературные» memories из 4 фактов в каждом entry — пускай модель пишет как хочет, эмбеддинг важнее ключевых слов.
- **Memory с историей**: к каждой Memory добавить `MemoryRevision[]` — как менялся `content` по неделям. При формировании промпта — last 3 ревизии вместо last single state.
- Менее «структурированные», более прозаичные memories. Не `{type, title, content}`, а просто текст 80-200 слов про одну тему.

### 🟠 Проблема №6. Нет редактирования — только полная регенерация

Кнопки [callbackQuery.ts]() предлагают: «короче / менее драматично / более литературно / более как я / перегенерить».

**Все эти кнопки делают полную перегенерацию всей страницы.**

Пользователь не может:
- Поправить один абзац.
- Добавить одну фразу.
- Удалить одно предложение.
- Изменить имя персонажа.
- Переписать заголовок.

Любая правка = играть в рулетку, надеясь, что новая версия будет лучше — но рискуя что-то ещё хуже сломается.

#### Почему это критично для мемуара

В дневнике это нормально (написал-забыл). В МЕМУАРЕ пользователь редактирует, потому что **правда важнее литературности**. Если AI поставил «я был спокоен», а пользователь был зол — он хочет это поправить. Сейчас невозможно.

#### Что нужно изменить

Минимум:
- Кнопка «✍ Подправить» → бот: «пришли исправление цитатой или новым абзацем» → AI получает `{previous_body, user_revision}` и интегрирует.
- Кнопка «Переписать заголовок» → отдельный мини-prompt только на title.
- Долгосрочно: web-редактор (LivingBook.tsx уже почти готов как читалка — добавить inline-edit).

### 🟠 Проблема №7. Структура книги — 52 разрозненных слота, без частей и без глав

#### Что есть

`Page` имеет `kind ∈ {WEEKLY, PROLOGUE}` и `chapterId` (но он почти всегда null). На web-превью [LivingBook.tsx:81-106](apps/web/src/components/LivingBook.tsx#L81-L106) разделение **только по месяцам** (просто декоративный divider «АПРЕЛЬ»). В PDF [renderPdf.ts:117-126](packages/renderer/src/renderPdf.ts#L117-L126) — TOC по месяцам, плоский список заголовков.

#### Чего нет

- Частей (Part I / Part II) — на год их обычно 3 («начало года», «середина», «к зиме»).
- Реальных глав, объединяющих 4-6 связанных недель.
- Эпилога (только пролог).
- Возможности у пользователя сказать «эти 5 страниц — про моё расставание, назови это главой».

#### Что пользователь видит

PDF-книга в декабре → **5.5"×8.25", обложка, потом 52 страницы заголовков подряд, разделённых только месяцем**. Это не чтение. Это база данных в PDF-обёртке.

Реальная книга: акт I (зима/начало) → cliffhanger → акт II (весна/основное действие) → акт III (лето/осень/трансформация) → эпилог. С названными главами.

#### Что нужно изменить

1. **Включить уже имеющуюся `Chapter` модель в работу.** Каждые 4-6 страниц фоновый job: «эти страницы образуют ли тематическое целое? назови, напиши 2-абзацное вступление». Сохранить как Chapter, привязать `chapterId` к Page.
2. **Дать пользователю manual control в web**: выделить страницы → «объединить в главу» → ввести название.
3. **Финальный pass перед PDF**: AI смотрит все страницы и предлагает разделение на 3 части и сквозной заголовок каждой.
4. В PDF: cover → Part I title page → Chapter 1 → pages 1-5 → Chapter 2 → pages 6-10 → Part II ...

### 🟠 Проблема №8. «52 недели» — клетка для продукта

Жёсткое allowance ровно 52 записи в год создаёт эффекты:

- **Гонка**: пользователь чувствует, что ДОЛЖЕН написать каждую неделю. Записи становятся обязательными, теряют смысл.
- **Стыд**: пропустил неделю → catchup-сообщение → виноват → вообще не пишу.
- **Заполнение из обязательства**: «писал что-то про работу, потому что надо запись».
- **Free-tier 4 записи = 1 месяц**, потом paywall. Слишком быстро для habit-formation.

#### Что нужно изменить

- **Переименовать «Запись N из 52»** → «Страница N» (без потолка). 52 — это эстетический ориентир в обложке, не лимит контента.
- Принимать **2-3 записи за неделю** если в неделе случилось много, и **0 записей** некоторые недели — без чувства провала.
- Ввести «спокойные напоминания» которые не заставляют (вместо «давай писать!» → «привет, на кухне за окном уже декабрь — что-то останавливалось в эту неделю?»).
- Free-tier: 8 записей (≈ 2 месяца), чтобы пользователь успел почувствовать ценность перед paywall.

### 🟡 Проблема №9. Нет отдельного уровня «жизненной хронологии»

Интейк собирает «БЫЛО до этого года». Weekly entries — «СЕЙЧАС, эта неделя». Между ними — пропасть.

Пользователь не может рассказать про:
- Прошлый год
- 5 лет назад
- Детство в подробностях (только Q1 онбординга, очень общо)
- Встречу с супругом 10 лет назад
- Учебу в ВУЗе

В реальном мемуаре это **главный материал**. LifeBookAI запирается в «here and now».

#### Что нужно изменить

- Дополнительная команда `/memory` или «📓 Recall»: пользователь говорит «я хочу записать что-то из прошлого». Бот спрашивает «когда это было?» и пишет страницу с метаданными `{periodStart: 2018-06}`.
- В LivingBook эти retrospective-страницы лежат на правильном месте по времени, не по дате создания.
- Пролог тогда становится не «один из 5 актов» а «первая партия retrospective-страниц», которую пользователь дополняет в любой момент.

### 🟡 Проблема №10. Пролог пишется один раз и не растёт

5 страниц пролога генерятся в момент онбординга и фиксируются. Никогда не пересматриваются.

Пользователь:
- через 3 месяца использования сильно лучше понимает свою историю,
- интейк-ответы у него были скомканные,
- хочет переписать пролог исходя из накопленного материала.

Сейчас это невозможно.

#### Что нужно изменить

- Команда `/prologue` для пересмотра пролога.
- Через 13 недель (квартал) — автоматическое предложение «хочешь, я перепишу пролог теперь, когда у нас уже 13 страниц?». Пересборка на расширенном корпусе.
- В web — inline-edit пролога.

### 🟡 Проблема №11. AI-обложка — одиночный эксперимент

`generateCover` ([packages/ai/src/generateCover.ts]) делает ОДНУ попытку через `gpt-image-1`, сохраняет результат, ставит `coverVersion=milestone`. Если она вышла плохой — пользователь видит её до следующего milestone.

- Нет вариантов на выбор (3-4 черновика).
- Нет переролла «эта не очень, попробуй ещё».
- Нет редактирования промпта.
- Стиль обложки задан в 1 системном промпте на всех — без учёта голоса книги.

#### Что нужно изменить

- На milestone генерить 3 варианта параллельно (`n: 3`), показывать пользователю в Telegram inline-keyboard «Variant A / B / C / Reroll».
- Хранить `Book.coverPromptUsed` — пусть пользователь увидит и поправит. Pro-фича.
- Стиль обложки увязать с `User.writingStyle` (calm warm → тёплая палитра, ironic → монохром, …).

### 🟡 Проблема №12. PDF не выглядит как книга

[renderPdf.ts](packages/renderer/src/renderPdf.ts):

- 5.5"×8.25" — нестандартный mass-market. Канон memoir-trade — 6"×9" или 5.5"×8.5".
- Только встроенные шрифты pdfkit (Times-Roman, Times-Bold, Times-Italic). Нет нормальной типографики.
- Нет нумерации страниц.
- Нет колонтитулов (running header).
- Нет drop caps (буквица в начале главы).
- Нет TOC с номерами страниц — просто список заголовков под месяцами.
- AI-обложка растягивается до 5.5×8.25 (`width: PAGE_W, height: PAGE_H` без `fit:`), а сама часто 1024×1024 → искажение.
- Каждая запись на новой странице, даже если запись 80 слов → много воздуха.

#### Что нужно изменить

- Зарегистрировать пристойный книжный шрифт через `doc.registerFont()` (EB Garamond, Crimson, Lora — все с Cyrillic-coverage).
- Колонтитулы: чёт/нечет, левая = название книги, правая = текущий месяц.
- Нумерация страниц.
- Drop cap на первой букве каждой записи (или каждой главы — лучше).
- Bleed/trim для KDP-готовности (как Pro+ опция).
- AI-обложка: `fit: [PAGE_W, PAGE_H]` или `cover` с правильным cropping.

### 🟡 Проблема №13. Уточняющие вопросы не используют контекст

[generateQuestions.ts](packages/ai/src/generateQuestions.ts) на вход получает: текущий ввод + 4 заголовка прошлого + до 6 memories. Опять — **не получает прозы**.

Поэтому уточнения часто звучат генерически: «Кто был рядом? Что изменилось?». Те самые вопросы, от которых system prompt сам же и предостерегает («Never ask about feelings in the abstract»).

Реальный биографа спросит: «Ты упоминал в апрельской записи про разговор с мамой на той же кухне — это было до или после?». Сейчас невозможно — модель не видит апрельскую запись.

#### Что нужно изменить

То же лекарство, что и для самой генерации: подавать тела 1-2 семантически близких страниц прошлого. Тогда вопросы станут конкретными.

### 🟡 Проблема №14. Memory как «лента в бот» не существует

Команда `/memories` ([commands/memories.ts]()) показывает всё списком, сгруппированным по типу. Пользователь не может:
- Посмотреть «какие memories AI вытащил из последней записи?» (для коррекции).
- Понять «откуда AI знает, что у меня брат Денис» (нет привязки к источнику кроме `sourceChapterId` в схеме, который для Memory.category=EXTRACTED почти не используется).
- Доверять корпусу — потому что не видит, что в нём.

#### Что нужно изменить

- После каждой записи: short telegram-сообщение «📌 Я запомнил: брат Денис, Калининград, страх быть забытым». Кнопка «✏ исправить» / «🗑 удалить».
- В `/memories` фильтр по дате создания и обратная ссылка на страницу-источник.

### 🟡 Проблема №15. Нет связи между голосом и текстом записи

Пользователь шлёт голосовое → бот транскрибирует → дальше всё работает с транскриптом. Но:
- **Транскрипт не показывается пользователю до того, как страница пишется.** Если whisper ошибся — пользователь видит ошибку только в финальной странице, и непонятно, ошибка модели или транскрипции.
- **Голосовой и текстовый интейки не различаются** — но они дают разный материал. Голосовой обычно богаче деталями, но менее структурирован.

#### Что нужно изменить

- После транскрипции — показывать пользователю transcript: «Я услышал так: [текст]. Всё ок? — ✅ Да / ✏ Поправлю».
- Если правит → правка идёт в `Entry.transcript` ДО генерации страницы.

## 5. Что я бы делал ПЕРВЫМИ ТРЕМЯ ШАГАМИ

Если у тебя один спринт на улучшение продукта — вот мой выбор:

### Шаг 1 (1 неделя): Sliding-window прозы + семантический ретривер
Это лекарство от Проблемы №1 (52 разрозненные вижнетки).

- Добавить в `generateEntry` поле `recentBodies` — full text 2 последних страниц + 5 страниц пролога.
- Добавить таблицу `PageEmbedding { pageId, vector }`. На каждое создание Page — фоновый job рассчитывает embedding (`text-embedding-3-small`).
- В `generateEntry` перед промптом — семантический поиск top-3 страниц по входу пользователя; их тела идут в новое поле `relatedBodies` промпта.
- Обновить системный промпт: явно сказать модели, что в `relatedBodies` лежат сцены, к которым НУЖНО сделать конкретную ссылку (имя, место, фраза — но не пересказ).

**Эффект:** пользователь к 5-й странице начинает чувствовать, что бот *помнит* — потому что в страницах действительно появляются конкретные переклички.

### Шаг 2 (3-5 дней): Карточка + чат-доставка
Это лекарство от Проблемы №2 (карточка обрезает текст).

- Уменьшить `BODY_MAX_LINES` на карточке с 30 до 8.
- Увеличить body font-size с 22 до 44pt.
- Карточка теперь показывает: title + цитата + 5-7 строк начала сцены.
- ОТДЕЛЬНЫМ сообщением `ctx.reply(body)` отправлять полный текст с сохранением `\n\n`.
- Подпись карточки меняется с «Запись N из 52» на «Страница N». Кнопка «Открыть мою книгу» остаётся.

**Эффект:** пользователь открывает Telegram, видит красивую обложку-плакат + сразу под ней читаемый полный текст. Карточку можно репостнуть, текст — прочитать.

### Шаг 3 (1 неделя): Главы из 4-5 страниц
Это лекарство от Проблем №7 и №16.

- Включить `Chapter` модель.
- Раз в неделю фоновый job: смотрит все unconnected `Page`-страницы (`chapterId IS NULL`). Если их ≥ 4 — группирует по теме (LLM-кластеризация по тегам/моодам/embeddings). Создаёт `Chapter` с AI-сгенерированным title + 2-3 sentence intro. Ставит `Page.chapterId`.
- Web-превью [LivingBook.tsx](apps/web/src/components/LivingBook.tsx) перерисовывается: divider не «МЕСЯЦ», а **«ГЛАВА: <title>»** + chapter intro прозой.
- PDF: TOC по главам с номерами страниц, не по месяцам.

**Эффект:** PDF в декабре выглядит **как книга**, а не как выгрузка из дневника.

После этих 3 шагов продукт перестаёт быть «дневниковым приложением с красивой каёмкой» и становится «AI-биографом».
